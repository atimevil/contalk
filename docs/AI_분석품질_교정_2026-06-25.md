# AI 위험 분류 오탐 교정 — before/after

작성일 2026-06-25 · 대상 `backend/ai/classifier.py`, `backend/ai/clause_parser.py`, `backend/app/services/contract_service.py`, `frontend/src/components/RiskBadge.tsx`

## 문제 (사용자 제보로 발견)

정상으로 설계한 표준계약서(01번)를 분석했더니 표준 양식의 **보호 조항**(제3·4·6·7·9조)이 🚨위험으로 분류됐다. 설명문은 그럴듯했지만 **명백한 오탐**이다 — 표준계약서의 정형 조항은 법적 보호의 기준선이라 정상이어야 한다.

## 판단 기준

- **정상 계약서(01·02·05) = 오탐 테스트**: 거의 전부 정상이어야 통과(정밀도).
- **위험 계약서(03·04·06·07·08) = 탐지 테스트**: 불리 특약이 잡혀야 통과(재현율).
- "설명이 그럴듯한가"가 아니라 **정답 라벨과 대조**해서 판단한다.

## 측정 결과 (test_contracts 8건, 룰 경로 재현)

| | 수정 전 | 수정 후 |
|---|---|---|
| 문서 등급 정확도 | **3/8** | **8/8** |
| 정상 3건 오탐 | 제4·제9조 등 매 계약서 위험 표시 | **오탐 0** (전부 정상) |
| 위험 5건 탐지 | 전부 등급 safe(미탐) | **전부 high**, 개별 특약까지 표시 |
| 계약유형 감지 | 8/8 | 8/8 |

## 원인과 교정

1. **표준조항 오탐** — 문맥을 못 보는 키워드 규칙이 표준 보호 문구에 발화(`원상복구`→제9조, `임대인 동의 없이`→제4조).
   → `classifier.py`에 **표준계약서 안전 베이스라인**(`_STANDARD_SAFE_PATTERNS`) 추가. 표준 정형·보호 조항(제3·4·6·7·8·9조 문구)은 모델/규칙보다 우선해 정상으로 고정. 표준에서 **벗어난 특약만** 위험 분류가 유지된다. (모델 앞단에 두어 모델 오탐도 억제)

2. **위험 특약 미탐** — 여러 특약이 "특약사항" 한 덩어리로 묶여 위험도가 희석됨 + 일부 불리 문구가 규칙에 안 걸림.
   → `clause_parser.py`: 특약 블록을 **번호별 개별 특약으로 분리**(`_split_special`).
   → `classifier.py`: 누락 위험 규칙 보강 — 1회/즉시 연체 해지, 최고 없이 해지, 보증금 반환 지연 면책, 동의 없이 임의 매도, 광범위 수선 전가, 중도해지 위약금, 별도 부담금·일방적 차임 인상, 전입신고/확정일자 차단(치명), 근저당 추가설정(표현 변형 대응).

3. **등급식** — 특약을 쪼개니 분모가 커져 비율 기준이 잘 안 올라감.
   → `contract_service._compute_risk_level`: 표준 베이스라인으로 정상 medium이 0에 수렴하므로 **개수 기반 승격 복원**(medium≥2 → 위험, medium==1 → 주의).

4. **표시 혼동** — `RiskBadge`가 medium·high를 동일하게 🚨위험으로 표시.
   → high=🚨고위험(red), medium=🟠위험(orange), caution=⚠️주의, safe=✅정상으로 분리.

## 재측정 방법 (실제 모델 포함)

```bash
# 문서 등급 정확도
docker compose run --rm --no-deps -w /work backend python scripts/measure_ai_quality.py
# 조항별 + test_contracts(한글 라벨) — 신규 하니스
docker compose run --rm --no-deps -w /work backend python scripts/measure_clause_risk.py
```

> 주의: 본 표의 수치는 룰 경로를 그대로 재현해 측정한 값이다. 실제 서비스는 KLUE-RoBERTa 모델 + 규칙 하이브리드이므로, 위 명령으로 모델 경로까지 포함한 최종 수치를 한 번 확인하길 권장한다. 규칙·특약분리·등급식·베이스라인은 두 경로에 공통 적용된다.
